## 1. 結論

AGENTS.mdのベストプラクティスは、**AIコーディングエージェントに対して、リポジトリ内での作業ルール・検証方法・禁止事項・完了条件を明確に伝えること**である。

特にCodex向けのAGENTS.mdは、長い方針書ではなく、**短く・具体的で・実行可能なリポジトリ契約**として設計するのが望ましい。調査資料でも、重要コマンド、リポジトリ構造、制約、完了条件、検証方法を明記し、抽象論より実例を優先することが有効とされている。

---

## 2. 理由

生成AIコーディングエージェントは、自然言語の曖昧な指示だけでも実装できるが、そのままでは以下の問題が起きやすい。

* リポジトリ構造を誤解する
* 実行すべきテストやLintを省略する
* 変更してはいけないファイルを編集する
* 依存関係を不要に追加する
* 実装だけして検証を行わない
* 人間がコードの意図を説明できないまま作業が進む

そのため、AGENTS.mdには「何をするか」だけでなく、**どう検証するか**と**どこまで触ってよいか**を明記する必要がある。調査資料でも、良いAGENTS.mdの構成要素として、repo layout、実行方法、build/test/lint、制約、done criteriaが重要とされている。

## 3. 具体案

````md
# AGENTS.mdのベストプラクティス

## 1. AGENTS.mdの目的

AGENTS.mdは、CodexやClaude Codeなどの生成AIコーディングエージェントに対して、リポジトリ内での作業ルールを伝えるためのファイルである。

このファイルの目的は、AIによる実装・修正・検証作業を安定させ、作業結果の再現性と品質を高めることである。

AGENTS.mdは、人間向けの長い説明書ではなく、AIエージェントが作業時に参照する「リポジトリ契約」として設計する。

## 2. 基本方針

良いAGENTS.mdは、以下の特徴を持つ。

- 短く具体的である
- 実行可能なコマンドが書かれている
- リポジトリ構造が明記されている
- 変更してよい範囲と禁止事項が明確である
- テスト・Lint・型チェックなどの検証方法が書かれている
- 完了条件が明確である
- AIが作業後に何を報告すべきかが決まっている

逆に、以下のようなAGENTS.mdは避けるべきである。

- 長すぎる方針書
- 抽象的な精神論
- 頻繁に変わる作業メモ
- 秘密情報を含む内容
- 実行できない曖昧な指示
- 検証方法が書かれていないルール

## 3. AGENTS.mdに書くべき内容

### 3.1 リポジトリの目的

リポジトリが何を目的としたものかを最初に書く。

例:

- このリポジトリは開発ハーネスを構築するためのものである
- このリポジトリはAIコーディングエージェントの実装結果を比較・記録するためのものである
- このリポジトリは実装力・設計理解・デバッグ力の維持を目的とする

目的を書くことで、AIエージェントが不要な機能追加や設計変更を避けやすくなる。

### 3.2 リポジトリ構造

主要なディレクトリと役割を明記する。

例:

- `docs/`
  - 設計書、仕様書、要件定義、研究メモを置く
- `scripts/`
  - 実行補助スクリプト、検証スクリプト、ログ収集スクリプトを置く
- `toolbox/`
  - 配備対象のテンプレートや設定ファイルを置く
- `.github/workflows/`
  - GitHub Actionsのワークフローを置く
- `ai_log/`
  - AI作業ログ、調査メモ、比較検証ログを置く

リポジトリ構造を書くことで、AIが作業対象を誤りにくくなる。

### 3.3 実行コマンド

AIエージェントが実行すべきコマンドを明記する。

例:

```bash
# install
npm install

# test
npm test

# lint
npm run lint

# typecheck
npm run typecheck
````

実コマンドが未定の場合は、無理に推測せずTODOとして残す。

```bash
# test
TODO

# lint
TODO

# typecheck
TODO
```

コマンドはできるだけ正確に書く。
`テストして` のような曖昧な表現ではなく、`npm test` や `pytest -q` のように実行可能な形で書く。

### 3.4 変更してよい範囲

AIが変更してよい範囲を明確にする。

例:

* `docs/` は設計書・仕様書の更新に使用してよい
* `scripts/` は検証用スクリプトの追加・修正に使用してよい
* `toolbox/` は配備対象ファイルの更新に使用してよい
* テスト用ファイルは必要に応じて追加してよい

### 3.5 変更に注意が必要な範囲

影響が大きい場所は、変更理由と検証方法を求める。

例:

* `.github/workflows/`

  * CI/CDに影響するため、変更理由と検証方法を明記する
* 依存関係ファイル

  * `package.json`
  * `package-lock.json`
  * `pyproject.toml`
  * `requirements.txt`
* 設計方針に関わるドキュメント

  * 変更前後の意図を説明する

### 3.6 禁止事項

AIに実行してほしくないことを明確に書く。

例:

* `.env` を変更しない
* 秘密情報をコード・ログ・ドキュメントに書かない
* `main` に直接pushしない
* `git reset --hard` や force push を勝手に実行しない
* 不要な依存関係を追加しない
* 既存の設計意図を無視して全面改修しない
* テスト未実行なのに「問題なし」と断定しない
* 根拠なく仕様を追加しない

禁止事項は、抽象的に「安全に作業する」と書くよりも、具体的な操作名やファイル名で書く方がよい。

### 3.7 完了条件

作業がどこまで進めば完了なのかを明記する。

例:

```md
## Done when

作業は以下を満たしたときに完了とする。

- 目的に対する変更が完了している
- 関連ファイルの差分を確認している
- 実行可能な検証コマンドを実行している
- 検証できない場合は理由を明記している
- 変更内容を日本語で説明できる
- 必要に応じてdocsまたはai_logを更新している
```

Done whenを入れることで、AIが実装だけで作業を終えず、検証・差分確認・報告まで行いやすくなる。

### 3.8 作業後の報告項目

AIが作業後に何を報告するかを決めておく。

例:

```md
## Report format

作業後は以下を報告する。

- 変更したファイル
- 変更内容
- 実行した検証
- 検証結果
- 残課題
- 人間が確認すべき点
```

これにより、人間がレビューしやすくなる。

### 3.9 Git / PRルール

Git操作やPR運用のルールを書く。

例:

* `main` へ直接pushしない
* 変更は原則PR経由で反映する
* 破壊的操作は明示依頼がある場合のみ実行する
* PR本文には以下を書く

```md
## 目的

## 変更点

## 検証結果

## 影響範囲

## 人間が確認すべき点

## 残課題
```

### 3.10 ログ運用

研究や検証に使う場合は、AI作業ログのルールを明記する。

例:

```md
## ログ運用

作業記録の正本はPR本文とする。

ただし、以下は `ai_log/` に残してよい。

- PR化しない調査メモ
- Codexとの試行錯誤ログ
- 比較検証の一次ログ
- 失敗したプロンプト
- 失敗した実装方針
- 実験条件のメモ
```

ログには、実行日時、使用エージェント、入力プロンプト、実行内容、変更ファイル、実行コマンド、結果、失敗点、人間が判断したことを残すとよい。

## 4. AGENTS.mdに書かない方がよい内容

AGENTS.mdには、すべてを詰め込まない方がよい。

書かない方がよい内容は以下である。

* 長大な手順書
* 毎回変わる作業メモ
* 個別タスクの詳細すぎる計画
* 秘密情報
* APIキー
* パスワード
* 環境固有の一時情報
* 抽象的すぎる開発思想
* AIが実行できない曖昧な指示

AGENTS.mdは、恒久的なリポジトリルールに絞るべきである。

## 5. 他ファイルとの役割分担

AGENTS.mdにすべてを書くと肥大化し、AIが重要な指示を見失いやすくなる。

そのため、以下のように役割分担する。

| ファイル                 | 役割              |
| -------------------- | --------------- |
| `AGENTS.md`          | リポジトリ全体の恒久ルール   |
| `SKILL.md`           | 繰り返し使う作業手順      |
| `PLANS.md`           | 長時間・高難度タスクの実行計画 |
| `.codex/config.toml` | Codexの実行ポリシー    |
| `rules/`             | 強制したい操作ルール      |
| `hooks`              | 自動検査・ログ・ガード処理   |

AGENTS.mdはリポジトリ契約、SKILL.mdは反復ワークフロー、PLANS.mdは複雑作業の計画、configは実行ポリシーとして分けると保守しやすい。

````

## 4. 実力維持の観点

本研究では、AGENTS.mdを単なる自動化ルールとして扱うのではなく、**AIを使いながら人間の実装力を維持するための仕組み**として位置づけるべきである。

研究計画でも、AI活用により開発速度は向上する一方で、コードの中身を十分に理解しないまま実装が進む危険があり、それが設計力・実装力・デバッグ力の低下につながると整理されている。:contentReference[oaicite:2]{index=2}

そのため、AGENTS.mdには次のような理解確認項目を入れるとよい。

```md
## Understanding Check

作業後に以下を説明する。

- 変更したファイル
- なぜ変更したか
- 実装の要点
- 実行した検証
- 失敗時に確認する場所
- 人間が追加で確認すべき点
````
