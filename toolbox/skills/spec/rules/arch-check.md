# Arch-Check Phase Rules

## 目的

設計レビュー。design.md 完成後に、設計原則との整合性・最適化余地・技術リスクを独立した視点で評価する。

## 必須

**`/senior-architect` スキルを使用する**

requirements.md + design.md + 既存コードベースの全文を `/senior-architect` に入力として渡し、独立した視点で設計をレビューする。

## 入力

- requirements.md
- design.md（+ adr.md がある場合はそれも含む）
- 既存コードベース

## 出力

- arch-check.md

## チェック内容

### 1. 設計原則チェック

全項目に **PASS/FAIL** 判定を付ける。FAIL がある場合は具体的な改善案を記載。

- **YAGNI**: 本当に今必要な機能か。将来のための過剰設計がないか
- **KISS**: より単純な方法はないか。不必要な抽象化がないか
- **DRY**: 知識の重複がないか（ただし過度な DRY で結合度が上がっていないか）
- **SRP**: 各コンポーネントの責務が単一か
- **SOLID**: 依存関係は適切か（特に DIP: 抽象に依存しているか）
- **過剰設計**: 必要以上のパターン・レイヤー・抽象化がないか

### 2. 決定タイプ判断

design.md §6 の決定記録を検証:

- **Type 1（不可逆）**: DB選定、認証方式、外部API採用 → ADR が作成されているか確認
- **Type 2（可逆）**: 内部実装、UI詳細 → 過度に慎重になっていないか確認

### 3. トレードオフ分析

設計で選択された各方針について、定量的な比較を行う:

```markdown
| 選択肢 | レスポンス | 開発工数 | 運用コスト | 推奨 |
|--------|-----------|---------|-----------|------|
| Redis キャッシュ | 10ms | 2週間 | $50/月 | ✓ |
| PostgreSQL のみ | 50ms | 1週間 | $0 | |

**選定理由**: レスポンス要件 (< 30ms) を満たすため Redis を採用
```

### 4. パフォーマンス・最適化レビュー

- **DB アクセス頻度**: N+1 問題の可能性、バッチ化の余地
- **クエリ最適化**: インデックス設計の妥当性、不要な JOIN の有無
- **キャッシュ戦略**: キャッシュすべきデータの見落とし
- **API 呼び出し**: 外部 API のレート制限、リトライ戦略の妥当性
- **非同期化**: 同期処理で十分か、非同期にすべき箇所の有無

### 5. 既存システムとの整合性

- 既存アーキテクチャパターンとの一貫性
- 既存コードの命名規則・ディレクトリ構造との整合
- 既存テストパターンとの整合

### 6. 技術リスク

| リスク | 影響度 | 発生確率 | 軽減策 |
|--------|--------|---------|--------|
| 例: 外部API の可用性 | 高 | 中 | サーキットブレーカー導入 |

## 成功条件

- 設計原則チェック: 全項目に PASS/FAIL 判定あり
- Type 1 決定: すべて識別され、ADR が存在することを確認
- トレードオフ: 各選択肢のメリット/デメリットを定量化
- パフォーマンスレビュー: DB アクセス頻度・最適化項目を洗い出し済み
- 推奨アクション: 必須/推奨/検討に分類して提示

## 停止条件（優先度順）

| 優先度 | 条件 | 対応 |
|--------|------|------|
| CRITICAL | セキュリティ懸念あり | 即時停止、ユーザー確認 |
| CRITICAL | 既存アーキテクチャと非互換 | 停止、影響範囲を報告 |
| HIGH | Type 1 決定が 3 つ以上 | 停止、分割を提案 |
| HIGH | 技術的実現性が不明 | Spike 実施を提案 |
| MEDIUM | 設計原則チェックで FAIL | design.md の修正を要求 |

## design.md 修正フィードバックループ

arch-check で FAIL や重大な最適化指摘が出た場合:

1. arch-check.md の「推奨アクション」セクションに修正内容を記載
2. **必須アクション**（CRITICAL/HIGH）がある場合 → design.md を修正
3. 修正後、該当箇所のみ再チェック
4. 全必須アクションが解消されるまでループ

## 遷移条件 → test-spec

- 全設計原則チェック完了（FAIL があれば design.md 修正済み）
- Type 1 決定の ADR 存在を確認済み
- トレードオフ分析で推奨案が選定済み
- 必須の推奨アクションがすべて対応済み

✅ Good トレードオフ分析:
