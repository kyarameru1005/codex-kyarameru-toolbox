# Hearing Phase Rules

## 目的

要件ヒアリング質問生成。requirements 作成前にユーザーから必要情報を収集。

## 成功条件

- 5W1H（What, Why, Who, When, Where, How）すべてに回答がある
- 非機能要件（性能、セキュリティ、可用性）の確認が完了
- 制約条件（予算、期限、技術制約）が明確

## 停止条件

| 条件 | 対応 |
|------|------|
| 「不明」回答が 3 項目以上 | 追加ヒアリング提案 |
| セキュリティ要件が不明確 | 停止、ユーザー確認 |
| 既存システムとの関連が不明 | 停止、調査依頼 |

## 遷移条件 → requirements

- 全必須項目（5W1H）に回答あり
- 矛盾する要件がない
- 非機能要件の方向性が決定

## Good/Bad パターン

```markdown
❌ Bad:
  - What: 「ユーザー管理」
  - Why: 「必要だから」
  - 性能: 「速く」

✅ Good:
  - What: 「メールアドレスによるユーザー認証機能」
  - Why: 「既存SNS認証に加え、企業ユーザー向けにメール認証を提供するため」
  - 性能: 「同時接続 1000 ユーザー、レスポンス 200ms 以内」
```

## チェックリスト

- [ ] What: 何を作るか具体的に定義
- [ ] Why: ビジネス価値・目的が明確
- [ ] Who: ターゲットユーザーが特定
- [ ] When: リリース時期・マイルストーン
- [ ] Where: 動作環境・制約
- [ ] How: 大まかな実現方法
- [ ] 性能要件: 定量的な数値
- [ ] セキュリティ要件: 認証・認可・データ保護
- [ ] 可用性要件: SLA・復旧目標
