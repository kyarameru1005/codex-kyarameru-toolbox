# Audit Rules

## 目的

Feature 品質検証。Feature Agent の実装完了後に独立した視点で品質を評価し、PASS/WARN/FAIL の総合判定を下す。

## 入力

- Feature の specs/（requirements.md, design.md, test-spec.md, tasks.md）
- Feature の実装コード
- テスト実行結果

## 出力

- audit-report.md（チェック結果 + 総合判定 + 修正指示）

## チェック手順（コスト効率順）

低コストのチェックから順に実行し、CRITICAL FAIL を早期検出する:

### 1. テスト結果チェック

| 項目 | PASS | WARN | FAIL |
|------|------|------|------|
| 全テストが通過 | 全パス | - | 1 件以上失敗 |
| カバレッジ | 80% 以上 | 60〜80% | 60% 未満 |
| テスト修正履歴 | 正当な修正のみ | 軽微な仕様変更 | テストを無効化して通過 |

### 2. 静的解析チェック

| 項目 | PASS | WARN | FAIL |
|------|------|------|------|
| lint エラー | 0 件 | warning のみ | error あり |
| 型エラー | 0 件 | - | 1 件以上 |
| セキュリティスキャン | 問題なし | LOW 脆弱性 | MEDIUM 以上の脆弱性 |

### 3. 設計整合性チェック

| 項目 | PASS | WARN | FAIL |
|------|------|------|------|
| design.md との一致 | API・型が完全一致 | 軽微な差異（命名のみ） | 構造的な乖離 |
| requirements.md の充足 | 全要件を満たす | 非機能要件に未達 | 機能要件に未達 |
| スコープ逸脱 | なし | 軽微な追加実装 | 大幅なスコープ逸脱 |

### 4. コード品質チェック

| 項目 | PASS | WARN | FAIL |
|------|------|------|------|
| 既存パターンとの一貫性 | 一貫している | 軽微な逸脱 | 独自パターンを導入 |
| エラーハンドリング | 適切に処理 | 一部未処理 | エラーを握りつぶし |
| ハードコード | なし | 設定値が 1〜2 箇所 | 秘匿情報のハードコード |

### 5. 依存関係チェック

| 項目 | PASS | WARN | FAIL |
|------|------|------|------|
| 依存 Feature の API 使用 | design.md 通り | バージョン差異 | 未定義 API を使用 |
| 新規依存ライブラリ | なし or 承認済み | 軽量ライブラリ追加 | 大規模依存追加 |
| 循環依存 | なし | - | 循環あり |

### 6. ドキュメントチェック

| 項目 | PASS | WARN | FAIL |
|------|------|------|------|
| ssot-updates.md | 変更差分が記載 | 不完全 | 未作成 |
| コードコメント | 適切 | 不足 | 誤解を招くコメント |
| CHANGELOG | 更新済み | - | 未更新（必須の場合） |

## 総合判定ルール

| 条件 | 総合判定 |
|------|---------|
| FAIL が 1 件以上（カテゴリ不問） | **FAIL** |
| FAIL なし、WARN が 1 件以上 | **WARN** |
| 全カテゴリ PASS | **PASS** |

### CRITICAL FAIL（即時エスカレーション）

以下は通常の FAIL より優先度が高く、即座にエスカレーションする:

- セキュリティスキャンで HIGH 以上の脆弱性
- テストを無効化して通過させている
- 秘匿情報のハードコード
- 循環依存の導入

## 修正優先度

| 優先度 | 基準 | 対応期限 |
|--------|------|---------|
| CRITICAL | セキュリティ・データ整合性に影響 | 即時修正必須 |
| HIGH | 機能要件未達・テスト失敗 | Merge 前に修正必須 |
| MEDIUM | コード品質・設計乖離 | Merge 前に修正推奨 |
| LOW | ドキュメント不足・命名改善 | 次回対応可 |

## 修正フィードバックループ

### FAIL 判定時

1. audit-report.md に修正指示を記載（優先度付き）
2. Feature Agent を修正指示付きで再起動（resume）
3. 修正完了後、**該当カテゴリのみ**再 Audit
4. 全カテゴリ PASS or WARN のみ → 完了

```
Feature Agent 完了
  → Audit Agent 起動
    → FAIL → 修正指示を Feature Agent に渡す
      → Feature Agent 修正実行
        → 再 Audit（該当カテゴリのみ）
          → PASS → 完了
          → FAIL → 再修正（最大 3 回）
            → 3 回失敗 → エスカレーション
```

### WARN 判定時

- WARN のみの場合は **PASS 扱い**で Merge 可能
- ただし WARN 内容を統合 PR の注意事項として記録

## 成功条件

- 6 カテゴリすべてのチェックが完了
- 総合判定が PASS or WARN
- FAIL がある場合は修正フィードバックループで解消済み
- audit-report.md が作成済み

## 停止条件

| 条件 | 対応 |
|------|------|
| CRITICAL FAIL 検出 | 即時停止、エスカレーション |
| 修正フィードバックループ 3 回失敗 | 停止、ユーザー確認 |
| テスト環境が利用不可 | 停止、環境修復を要求 |
