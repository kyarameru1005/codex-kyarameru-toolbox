# Requirements Phase Rules

## 目的

hearing.md の回答を基に、検証可能な要件定義を作成。

## 入力

- hearing.md（ヒアリング結果）

## 出力

- requirements.md
- 03_USE_CASES.md 更新

## 成功条件

- ユースケース: 最低 1 つ以上定義
- 受入条件: 各ユースケースに 1 つ以上の検証可能な条件
- 非機能要件: 性能・セキュリティ・可用性を定量化
- 03_USE_CASES.md に UC 追記完了

## 停止条件

| 条件 | 対応 |
|------|------|
| hearing.md 必須項目が未回答 | hearing に戻る |
| 矛盾する要件を検出 | ユーザー確認 |
| 実現不可能な要件 | ユーザー確認、代替案提示 |

## 遷移条件 → design

- 全ユースケースに受入条件あり
- 非機能要件が定量化されている
- 03_USE_CASES.md 更新完了

## Good/Bad パターン

```markdown
❌ Bad:
  - 「高速に動作すること」
  - 「セキュリティを考慮する」
  - 「ユーザーが使いやすい」

✅ Good:
  - 「レスポンスタイム: 95%ile < 200ms」
  - 「認証: OAuth 2.0 + PKCE、セッション有効期限 24h」
  - 「UC-001: ユーザーはメールとパスワードでログインできる」
    - 受入条件: 正しい認証情報で 3 秒以内にダッシュボードへ遷移
```

## UC 採番ルール

`specs/03_USE_CASES.md` から対象ドメインの最大番号+1

## チェックリスト

- [ ] ユースケースに UC-XXX 形式の ID 付与
- [ ] 各 UC に検証可能な受入条件
- [ ] 性能要件に具体的な数値（レスポンス、スループット）
- [ ] セキュリティ要件に具体的な方式（認証方式、暗号化）
- [ ] 可用性要件に SLA 数値（稼働率、RTO/RPO）
- [ ] 03_USE_CASES.md に新規 UC を追記
