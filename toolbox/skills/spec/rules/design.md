# Design Phase Rules

## 目的

詳細設計書を作成する。requirements.md とコードベース調査結果をもとに、実装可能なレベルまで設計を詳細化する。

## 入力

- requirements.md
- コードベース調査結果（Agent(Explore) の出力）

## 出力

- design.md
- adr.md（Type 1 決定時）

## 必須項目（優先度順）

| 優先度 | 項目 | 必須条件 |
|--------|------|----------|
| CRITICAL | セキュリティ設計 | 認証・認可・データ保護の方式 |
| CRITICAL | エラーハンドリング | 異常系フロー、リトライ戦略 |
| HIGH | API 設計 | エンドポイント、リクエスト/レスポンス定義 |
| HIGH | データモデル | ER図またはテーブル定義 |
| HIGH | Type 1/Type 2 決定記録 | 各決定の根拠と選択肢を記載 |
| MEDIUM | シーケンス図 | 主要フローの処理順序 |
| MEDIUM | 状態遷移 | ステートマシン（該当時） |
| LOW | UI 詳細 | ワイヤーフレーム（任意） |

### Type 1/Type 2 決定

設計中に判明した決定事項を design.md §6 に記録する:

- **Type 1（不可逆）**: DB選定、認証方式、外部API採用 → ADR 作成必須
- **Type 2（可逆）**: 内部実装、UI詳細 → design.md 内で記録

**注意**: 設計原則チェック（YAGNI, KISS, SRP, SOLID）やトレードオフの定量分析は後続の arch-check フェーズで `/senior-architect` スキルを用いて実施する。design フェーズでは設計そのものに集中する。

## 成功条件

- CRITICAL 項目: 100% 定義
- HIGH 項目: 100% 定義
- MEDIUM 項目: 80% 以上定義
- Type 1 決定が識別され、ADR 作成完了

## 停止条件

| 優先度 | 条件 | 対応 |
|--------|------|------|
| CRITICAL | セキュリティ設計が定義不可 | 停止、ユーザー確認 |
| CRITICAL | 既存アーキテクチャと非互換 | 停止、影響範囲を報告 |
| HIGH | Type 1 決定が 3 つ以上 | 停止、分割を提案 |
| HIGH | 要件不足で設計不可 | requirements に戻る |
| MEDIUM | 技術的に複数案あり | 選択肢を記録し、arch-check で定量比較 |

## 遷移条件 → arch-check

- CRITICAL/HIGH 項目すべて完了
- ADR 作成完了（Type 1 決定がある場合）
- セキュリティ設計・エラーハンドリング設計が定義済み

## セキュリティ設計チェックリスト

- [ ] 認証方式（OAuth, JWT, Session 等）
- [ ] 認可方式（RBAC, ABAC 等）
- [ ] データ保護（暗号化、マスキング）
- [ ] 入力検証（バリデーション方針）
- [ ] 監査ログ（記録対象、保持期間）

## エラーハンドリングチェックリスト

- [ ] 異常系フローの定義
- [ ] リトライ戦略（回数、間隔）
- [ ] フォールバック方針
- [ ] ユーザーへのエラー表示方針
- [ ] ログ出力方針

## チェックリスト

- [ ] セキュリティ設計完了
- [ ] エラーハンドリング設計完了
- [ ] API エンドポイント定義
- [ ] データモデル定義
- [ ] シーケンス図作成
- [ ] Type 1/Type 2 決定を記録
- [ ] Type 1 決定の ADR 作成
