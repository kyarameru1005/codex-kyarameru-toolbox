# Refine Phase Rules

## 目的

既存の Product Backlog をリファインメントする。優先順位の見直し、見積もり修正、新規アイテム追加、完了アイテムの反映。

## 参照必須

- `~/.claude/skills/senior-architect/references/agile-architecture.md`（Product Backlog, Story Points）

## トリガー

以下のタイミングで `/spec refine` を実行:

| タイミング | 理由 |
|-----------|------|
| マイルストーン完了後 | 次のマイルストーンに向けた再計画 |
| `/spec retro` の後 | Retrospective の Action Items を反映 |
| 大きな要件変更時 | 優先順位の見直し |
| 見積もりが大幅にずれた時 | SP 基準の再校正 |

## 成功条件

- 完了アイテムが Done に更新されている
- 新規アイテムが追加されている（該当する場合）
- 優先順位が現在の状況を反映している
- SP 見積もりが最新の知見を反映している
- バックログサマリが再計算されている
- リファインメント履歴に記録されている

## 停止条件

| 条件 | 対応 |
|------|------|
| BACKLOG.md が存在しない | 停止、`/spec backlog` を先に実行 |
| 優先順位の判断がつかない | 停止、ユーザーに確認 |
| 大幅なスコープ変更 | 停止、ROADMAP の見直しを提案 |

## リファインメント手順

1. **現状確認**: BACKLOG.md の現在の状態を把握
2. **完了反映**: Done の PBI を更新
3. **新規追加**: Retrospective の Try や新規要件を PBI 化
4. **再優先順位**: 現在の状況に基づき並べ替え
5. **再見積もり**: 新しい知見に基づき SP を修正
6. **サマリ更新**: 総 SP、残り SP を再計算
7. **履歴記録**: リファインメント履歴に追記

## Good/Bad パターン

```markdown
Bad:
  - 完了した PBI がそのまま Ready
  - 3ヶ月前の見積もりを更新せず
  - 履歴の記載なし

Good:
  - Done: PBI-001 (user-auth)
  - 修正: PBI-003 SP:8→5 (実績から再見積もり)
  - 追加: PBI-010 error-handling (retro P#2 対応)
  - 履歴: "2026-01-28: v0.1完了に伴うリファインメント"
```
