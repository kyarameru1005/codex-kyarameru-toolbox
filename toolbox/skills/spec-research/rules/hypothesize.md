# Hypothesize ルール - 仮説生成フレームワーク

## 仮説生成プロセス

### Step 1: 観察データの分類

observations.md のデータを4象限に分類:

| | 良いトレンド | 悪いトレンド |
|---|---|---|
| **予想通り** | 強化（もっと伸ばす） | 修正（原因を特定） |
| **予想外** | 発見（なぜうまくいっているか） | 調査（新しい問題か） |

### Step 2: 仮説テンプレート

各仮説は以下の構造で記述:

```markdown
### H-{seq}: {タイトル}
- **観察**: {observations.md の該当データ}
- **仮説**: {なぜこうなっているか}
- **検証方法**: {どうすれば検証できるか}
- **期待される結果**: {仮説が正しければ何が起こるか}
- **ICE スコア**: I={1-10} C={1-10} E={1-10} → 合計={I*C*E}
- **ステータス**: NEW / TESTING / TESTED / REJECTED
```

### Step 3: ICE スコアリング

| 軸 | 1-3 (低) | 4-6 (中) | 7-10 (高) |
|---|---|---|---|
| **Impact** | 少数ユーザーに影響 | 主要フローに影響 | 全ユーザーの核心体験に影響 |
| **Confidence** | 推測のみ | 間接的証拠あり | 直接的データあり |
| **Ease** | 大規模リファクタ必要 | 数ファイル変更 | 1ファイル以内で完了 |

### Step 4: 過去の学びとの照合

`research/experiment-journal.md` を読み、以下をチェック:
- 同じ仮説を過去にテストしていないか → `[DUPLICATE]`
- 過去の実験の学びと矛盾しないか
- 過去の失敗パターンに該当しないか

### Step 5: 見落とし防止チェック

`spec-verify-matrix` の SFDIPOT 7軸で確認:
- **S**tructure: アーキテクチャ的な問題を見落としていないか
- **F**unction: 機能的な改善余地はないか
- **D**ata: データの扱い方に問題はないか
- **I**nterface: API/UI のインターフェースに問題はないか
- **P**latform: 環境依存の問題はないか
- **O**perations: 運用面の改善余地はないか
- **T**ime: 時間に関する問題はないか

## 仮説の制限

- 1回の実行で最大 5 件
- ICE 合計 < 27（各軸3未満）の仮説は生成しない
- セキュリティ・プライバシーに関わる仮説は `[REQUIRES_HUMAN]` マーク

## 出力フォーマット

```markdown
# Hypotheses - Updated {date}

## Active Hypotheses (ICE desc)

### H-001: オンボーディング step3 の離脱率改善
- 観察: step3 離脱率 45%（他step平均 15%）
- 仮説: 説明文が長すぎて認知負荷が高い
- 検証方法: 説明文を200字→80字に短縮してbefore/after比較
- 期待される結果: 離脱率 30% 以下に改善
- ICE: I=8 C=7 E=9 → 504
- ステータス: NEW

### H-002: ...

## Tested Hypotheses
### H-000: ... [TESTED] → 結果は experiment-journal 参照

## Rejected Hypotheses
### H-xxx: ... [REJECTED] → 理由: ...
```
