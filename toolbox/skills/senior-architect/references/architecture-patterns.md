# アーキテクチャパターン比較 (2025-2026)

## Monolith vs Modular Monolith vs Microservices

### 判断フローチャート

```
チームサイズ 10人以下?
  → Yes: Monolith or Modular Monolith
  → No: 独立したスケーリングが必要?
         → Yes: Microservices
         → No: Modular Monolith
```

### 比較表

| 観点 | Monolith | Modular Monolith | Microservices |
|------|----------|------------------|---------------|
| 初期開発速度 | 最速 | 速い | 遅い |
| 運用複雑性 | 低 | 低〜中 | 高 |
| スケーラビリティ | 限定的 | 中程度 | 高い |
| チーム独立性 | 低 | 中 | 高 |
| デバッグ容易性 | 容易 | 容易 | 困難 |
| 必要エンジニア数 | 1-2 ops | 1-2 ops | 2-4+ platform |

### 2025-2026 トレンド

- **42%の組織**が一度採用したマイクロサービスを統合に戻している (CNCF 2025)
- **モジュラーモノリス**が新規プロジェクトでの人気上昇中
- 理由: デバッグ複雑性、運用オーバーヘッド、ネットワークレイテンシ

### 実例

| 企業 | アーキテクチャ | 理由 |
|------|----------------|------|
| Shopify | Modular Monolith + 限定MS | コアはモノリス、checkout/fraud は MS |
| GitHub | Monolith (Rails) | 数百万ユーザーをモノリスで処理 |
| Uber | Microservices | グローバル規模、地域別要件 |

## Event-Driven Architecture

### いつ使うか

- 非同期処理が多い
- サービス間の疎結合が必要
- イベントの監査ログが必要
- リアルタイム処理が必要

### パターン

| パターン | 用途 |
|----------|------|
| Event Notification | 状態変更の通知 |
| Event-Carried State Transfer | データ同期 |
| Event Sourcing | 完全な履歴保持 |
| CQRS | 読み書き分離 |

### 注意点

- 結果整合性の理解が必要
- デバッグが複雑になりやすい
- 過剰適用に注意

## Clean / Hexagonal Architecture

### 共通原則

1. 依存の方向を外から内へ
2. ドメインロジックを中心に
3. インフラ詳細を外側に
4. インターフェースで境界を定義

### いつ使うか

- ドメインロジックが複雑
- 長期保守が必要
- テスタビリティ重視
- インフラ変更の可能性あり

### 注意点

- 小規模プロジェクトには過剰
- 初期の学習コストあり
- ボイラープレートが増える
