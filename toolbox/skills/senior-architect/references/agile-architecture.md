# アジャイルアーキテクチャガイド

## 概要

アジャイル開発におけるアーキテクチャは「事前に全て決める」のではなく、
「必要な時に必要な決定をする」アプローチを取る。

**原則**:
- 早くリリースして早くフィードバックを得る
- 変更を受け入れやすい設計
- 過剰設計を避ける

---

## 1. Emergent Architecture (創発的アーキテクチャ)

### 概要

アーキテクチャは事前に全て設計するのではなく、開発を進めながら徐々に形作られる。

### 前提条件

創発的アーキテクチャが機能するために必要なこと:
- 継続的なリファクタリング
- テスト駆動開発 (TDD)
- 頻繁なコードレビュー
- チーム全体のアーキテクチャ理解

### バランス

```
事前設計 ◄─────────────────────────────► 完全創発
  (Waterfall)                             (Chaos)
              ↑
            ここを目指す
         "Just Enough Architecture"
```

**事前に決めるべきこと**:
- 変更困難な決定（インフラ、主要言語、フレームワーク）
- チーム間の境界
- セキュリティ基盤

**後から決めてよいこと**:
- 詳細な設計
- ライブラリ選定
- 内部構造の最適化

---

## 2. Last Responsible Moment (最終責任時点)

### 概要

決定は「できる限り遅らせる」のではなく「必要になるまで待つ」。

### 原則

```
情報が少ない時の決定 → 間違いやすい
情報が多い時の決定 → 精度が上がる

ただし、遅すぎると機会を逃す
```

### 判断基準

| 決定タイミング | 条件 |
|----------------|------|
| **今すぐ決める** | 遅らせるとコストが大幅に増加する |
| **少し待つ** | より多くの情報が得られる見込みがある |
| **先送りする** | まだ情報が不十分で、待てる余裕がある |

### 実践

1. 決定が必要な項目をリストアップ
2. 各決定の「最終決定期限」を特定
3. 期限まで情報収集
4. 期限で決定

## 3. Type 1 / Type 2 決定 (Bezos フレームワーク)

### 概要

全ての決定が同じ重さではない。決定の可逆性によってアプローチを変える。

### 分類

| タイプ | 特徴 | アプローチ |
|--------|------|------------|
| **Type 1** | 不可逆、片道ドア | 慎重に、十分な検討を |
| **Type 2** | 可逆、両道ドア | 素早く、失敗したら戻す |

### Type 1 決定の例

- プログラミング言語の選定
- クラウドプロバイダーの選定
- データベース製品の選定
- マイクロサービス vs モノリス
- 外部公開 API のスキーマ

**アプローチ**:
- 十分な調査と比較
- ステークホルダーの合意
- ADR での記録
- PoC での検証

### Type 2 決定の例

- 内部ライブラリの選定
- コーディングスタイル
- テストフレームワーク
- 内部 API の設計
- UI コンポーネントの構成

**アプローチ**:
- 素早く決定
- 試してダメなら変更
- 過度な議論を避ける

## 4. Walking Skeleton パターン

### 概要

最小限のEnd-to-End機能を早期に実装し、アーキテクチャを検証する。

### 定義

```
Walking Skeleton = 最小限の機能 + 全レイヤーを貫通
```

**目的**:
- アーキテクチャの早期検証
- 技術的リスクの早期発見
- チームの学習
- デプロイパイプラインの確立

### 特徴

```
通常のMVP:      機能A + 機能B + 機能C（横に広い）
Walking Skeleton: 機能Aの最小実装（縦に深い）
                  UI → API → Domain → DB → 戻り
```

### 実装例

**ECサイトの場合**:
```
Walking Skeleton:
1. 商品一覧表示（1商品、ハードコード）
2. カートに追加
3. 注文確定（決済なし）
4. 注文完了画面

含まれるもの:
- フロントエンド基盤
- API基盤
- ドメインロジック最小限
- DB接続
- 認証の骨格
- CIパイプライン
- 本番デプロイ

含まれないもの:
- 全商品データ
- 決済連携
- 複雑なビジネスロジック
- 最適化
```

### 進め方

1. **Sprint 0-1**: Walking Skeleton 実装
2. **検証**: アーキテクチャは機能するか？
3. **調整**: 問題があれば早期に修正
4. **拡張**: 機能を追加していく

## 5. Sprint 0 決定事項

### 概要

最初のスプリント（Sprint 0）で決めるべき最小限の事項。

### チェックリスト

**必須（Sprint 0）**:

- [ ] **言語/ランタイム**: 変更困難なため早期決定
- [ ] **主要フレームワーク**: 基盤となるため早期決定
- [ ] **リポジトリ構成**: Monorepo vs Multirepo
- [ ] **CI/CD パイプライン**: 骨格を構築
- [ ] **デプロイ先環境**: クラウド、リージョン
- [ ] **認証方式の方針**: 自前 vs IDaaS
- [ ] **コーディング規約**: Linter/Formatter 設定

**Sprint 1-2 で決める**:

- [ ] データベース製品（PoC 後）
- [ ] キャッシュ戦略
- [ ] ログ/監視方針
- [ ] エラーハンドリング方針

**後から決める（必要になってから）**:

- [ ] 具体的なライブラリ選定
- [ ] 詳細な設計パターン
- [ ] 最適化手法

## 6. Fitness Functions (適応度関数)

### 概要

アーキテクチャの品質を継続的に計測・検証する仕組み。

### 定義

```
Fitness Function = アーキテクチャ特性を検証する自動テスト
```

### カテゴリ

| カテゴリ | 測定対象 | 例 |
|----------|----------|-----|
| **性能** | レスポンスタイム、スループット | p95 < 200ms |
| **構造** | 依存関係、モジュール結合度 | 循環依存がない |
| **セキュリティ** | 脆弱性、設定 | OWASP Top 10 対応 |
| **運用** | 可観測性、デプロイ頻度 | メトリクス取得率 100% |

### 実装例

**依存関係チェック (ArchUnit - Java)**:
```java
@Test
void domainShouldNotDependOnInfrastructure() {
    noClasses()
        .that().resideInAPackage("..domain..")
        .should().dependOnClassesThat()
        .resideInAPackage("..infrastructure..")
        .check(classes);
}
```

**性能チェック (CI)**:
```yaml
- name: Performance Test
  run: |
    k6 run load-test.js
    # p95 が 200ms を超えたら失敗
```

**セキュリティチェック**:
```yaml
- name: Security Scan
  run: |
    trivy image myapp:latest
    # 重大な脆弱性があれば失敗
```

### 運用

1. **定義**: 重要なアーキテクチャ特性を特定
2. **実装**: 自動テストとして実装
3. **CI 統合**: プルリクエストで自動実行
4. **閾値設定**: 許容範囲を定義
5. **アラート**: 閾値超過で通知

## 7. 漸進的な設計

### Set-Based Design

**概要**: 複数の選択肢を並行して検討し、徐々に絞り込む

```
Sprint 1: 選択肢 A, B, C を軽く試す
Sprint 2: A, B に絞る（C は情報不足で脱落）
Sprint 3: A を採用（B はコスト高で脱落）
```

**利点**:
- 早期のコミットメントを避ける
- より良い判断ができる
- リスクを分散

### Spike (技術調査)

**概要**: 時間を区切って技術的な不確実性を解消

**形式**:
```
タイトル: {調査対象}
目的: {何を明らかにするか}
タイムボックス: {X時間 or Xポイント}
成果物: {判断材料、PoC、レポート}
```

**例**:
```
タイトル: GraphQL vs REST 比較
目的: 当プロジェクトでどちらが適切か判断
タイムボックス: 8時間
成果物:
- 両方式での API 実装サンプル
- パフォーマンス比較
- 開発効率比較
- 推奨案
```

## 8. アジャイルにおける ADR

### 軽量 ADR

従来の ADR より軽量に、頻繁に作成:

```markdown
# ADR-{番号}: {タイトル}

**Date**: {YYYY-MM-DD}
**Status**: Proposed | Accepted | Superseded

## Context
{2-3文で背景}

## Decision
{1文で決定内容}

## Consequences
- (+) {良い影響}
- (-) {悪い影響}
```

### MADR (Markdown Any Decision Records)

GitHub で管理しやすい軽量形式:

```markdown
# {タイトル}

* Status: {status}
* Date: {YYYY-MM-DD}

## Context and Problem Statement
{問題の説明}

## Considered Options
* Option 1
* Option 2

## Decision Outcome
Chosen option: "{選択}", because {理由}.
```

## 9. チェックリスト

### Sprint 0 開始時

- [ ] Walking Skeleton の範囲を定義したか？
- [ ] Type 1 決定をリストアップしたか？
- [ ] CI/CD の骨格を構築したか？
- [ ] ADR プロセスを確立したか？

### 各スプリント開始時

- [ ] このスプリントで決定すべき事項は何か？
- [ ] 先送りできる決定はないか？
- [ ] 技術的リスクの高い項目はないか？

### 決定時

- [ ] Type 1 or Type 2 を判断したか？
- [ ] Type 1 なら十分な検討をしたか？
- [ ] ADR を作成したか？
- [ ] Fitness Function を追加すべきか？

### スプリント終了時

- [ ] アーキテクチャに問題は見つかったか？
- [ ] 技術的負債は記録したか？
- [ ] 次スプリントで決定すべき事項は何か？

## 10. アンチパターン

### Big Design Up Front (BDUF)

**症状**: 実装前に全てを設計しようとする
**問題**: 変更に対応できない、時間の無駄
**対策**: Walking Skeleton, 漸進的設計

### Analysis Paralysis

**症状**: 決定できずに分析を続ける
**問題**: 進捗しない、機会損失
**対策**: Type 1/2 判断、タイムボックス

### Ivory Tower Architecture

**症状**: 実装者と離れた場所で設計
**問題**: 実装困難、現実との乖離
**対策**: チーム全体でアーキテクチャ議論

### Gold Plating

**症状**: 必要以上の品質・機能を追加
**問題**: 時間の浪費、複雑化
**対策**: YAGNI, MVP 思考
