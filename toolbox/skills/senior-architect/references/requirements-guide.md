# 要件ヒアリング・定義ガイド

## 概要

要件定義は「何を作るか」を明確にするプロセス。
良い要件は、実装者が迷わず、テスターが検証でき、ステークホルダーが合意できる。

---

## 1. ヒアリング技法

### 質問のフレームワーク

**5W1H + So What**:
- What: 何を実現したいか？
- Why: なぜそれが必要か？（根本的な目的）
- Who: 誰が使うか？誰が影響を受けるか？
- When: いつ必要か？いつ使うか？
- Where: どこで使うか？（環境、コンテキスト）
- How: どのように使うか？
- **So What**: それが実現すると何が変わるか？

**深掘り質問**:
- 「もう少し詳しく教えてください」
- 「具体的な例を挙げていただけますか？」
- 「それがないと何が困りますか？」
- 「理想的にはどうあってほしいですか？」

---

### SPIN テクニック

| 種類 | 目的 | 例 |
|------|------|-----|
| Situation | 現状把握 | 「現在どのように業務を行っていますか？」 |
| Problem | 問題特定 | 「何か困っていることはありますか？」 |
| Implication | 影響理解 | 「その問題が続くとどうなりますか？」 |
| Need-payoff | 価値確認 | 「解決するとどんなメリットがありますか？」 |

---

### アクティブリスニング

- **パラフレーズ**: 「つまり〜ということですね」
- **要約**: 「ここまでの話をまとめると...」
- **確認**: 「〜という理解で合っていますか？」
- **沈黙**: 相手が考える時間を与える

## 2. 要件発見ワークショップ

### User Story Mapping

**概要**: ユーザーの行動を時系列で可視化し、MVPを特定

**構造**:
```
横軸: ユーザーの活動の流れ（時系列）
縦軸: 優先度（上が高い）

[活動1]      [活動2]      [活動3]
  |            |            |
[タスク1-1]  [タスク2-1]  [タスク3-1]  ← MVP ライン
[タスク1-2]  [タスク2-2]  [タスク3-2]  ← Release 2
[タスク1-3]              [タスク3-3]  ← Release 3
```

**進め方**:
1. ペルソナを定義
2. ユーザーの「活動」を左から右へ並べる
3. 各活動の「タスク」を下に展開
4. 優先度で並び替え
5. MVP ラインを引く

---

### Event Storming

**概要**: ドメインイベントを起点にシステムを理解

**付箋の色と意味**:
| 色 | 意味 | 例 |
|----|------|-----|
| オレンジ | ドメインイベント | 「注文が確定された」 |
| 青 | コマンド | 「注文を確定する」 |
| 黄 | アクター/ユーザー | 「購入者」 |
| 紫 | ポリシー/ルール | 「在庫があれば」 |
| ピンク | 外部システム | 「決済システム」 |
| 緑 | リードモデル | 「注文一覧画面」 |

**進め方**:
1. ドメインイベントを時系列で並べる
2. イベントを引き起こすコマンドを追加
3. コマンドを実行するアクターを追加
4. ビジネスルール（ポリシー）を追加
5. 集約境界を特定

---

### Example Mapping

**概要**: 具体例からルールを発見

**構造**:
```
[ユーザーストーリー] (黄)
    |
[ルール1] (青) ── [例1-1] (緑) [例1-2] (緑)
[ルール2] (青) ── [例2-1] (緑)
[質問] (赤)
```

**進め方**:
1. ユーザーストーリーを提示
2. 具体的な例を挙げる
3. 例からルールを抽出
4. 不明点は「質問」として記録

## 3. ユーザーストーリー

### フォーマット

**標準形式**:
```
As a [役割/ペルソナ]
I want [機能/行動]
So that [価値/目的]
```

**日本語形式**:
```
[役割]として
[機能]したい
なぜなら[価値]だから
```

---

### INVEST 原則

| 原則 | 意味 | チェック |
|------|------|----------|
| **I**ndependent | 独立している | 他のストーリーなしで実装可能か？ |
| **N**egotiable | 交渉可能 | 詳細は会話で決められるか？ |
| **V**aluable | 価値がある | ユーザーに価値を提供するか？ |
| **E**stimable | 見積可能 | チームが見積もれるか？ |
| **S**mall | 小さい | 1スプリントで完了するか？ |
| **T**estable | テスト可能 | 完了条件が明確か？ |

---

### 受入条件 (Acceptance Criteria)

**Gherkin 形式**:
```gherkin
Scenario: ログイン成功
  Given ユーザーがログイン画面にいる
  When 正しいメールとパスワードを入力する
  And ログインボタンをクリックする
  Then ダッシュボードに遷移する
  And ウェルカムメッセージが表示される
```

**チェックリスト形式**:
```
受入条件:
- [ ] 正しい認証情報でログインできる
- [ ] 間違ったパスワードでエラーメッセージが表示される
- [ ] 5回失敗でアカウントがロックされる
- [ ] ログイン後、リクエストしたページにリダイレクトされる
```

## 4. 非機能要件

### カテゴリと質問

| カテゴリ | 質問例 |
|----------|--------|
| **性能** | 許容レスポンスタイムは？同時ユーザー数は？ |
| **可用性** | ダウンタイムの許容範囲は？SLA 目標は？ |
| **スケーラビリティ** | 将来のユーザー数は？データ量の成長予測は？ |
| **セキュリティ** | 認証要件は？データ保護要件は？ |
| **運用** | 監視要件は？バックアップ頻度は？ |
| **互換性** | 対応ブラウザは？対応デバイスは？ |
| **法規制** | GDPR? 個人情報保護法? |

---

### 定量化の例

**悪い例**: 「高速であること」
**良い例**: 「95パーセンタイルで200ms以内」

**悪い例**: 「高可用性」
**良い例**: 「99.9% SLA（月間ダウンタイム43分以内）」

**悪い例**: 「スケーラブル」
**良い例**: 「1万同時接続に対応、年率50%成長を想定」

## 5. 優先順位付け

### MoSCoW メソッド

| 分類 | 意味 | 説明 |
|------|------|------|
| **Must** | 必須 | これがないとリリースできない |
| **Should** | 重要 | 強く望まれるが、回避策がある |
| **Could** | あれば良い | リソースがあれば対応 |
| **Won't** | 今回は対象外 | 今回は実装しない（将来検討） |

---

### Kano モデル

| 種類 | 説明 | 例 |
|------|------|-----|
| **当たり前品質** | なくて当然、あっても満足度上がらない | ログイン機能 |
| **一元的品質** | 充足度に比例して満足度上昇 | 検索速度 |
| **魅力的品質** | なくても不満なし、あると大満足 | AIレコメンド |
| **無関心品質** | あってもなくても影響なし | - |
| **逆品質** | あると不満、ないと満足 | 過剰な通知 |

**適用**:
- 当たり前品質: 最低限実装
- 一元的品質: 競合との差別化ポイント
- 魅力的品質: MVP後のリリースで検討

---

### RICE スコア

```
RICE = (Reach × Impact × Confidence) / Effort
```

| 要素 | 説明 | スケール |
|------|------|----------|
| Reach | 影響を受けるユーザー数 | 実数 |
| Impact | インパクト | 0.25/0.5/1/2/3 |
| Confidence | 確信度 | 50%/80%/100% |
| Effort | 工数 | 人週 |

## 6. DDD との連携

### ユビキタス言語

**定義**: 開発者とドメインエキスパートが共有する用語

**作成手順**:
1. ドメインエキスパートの言葉を記録
2. 曖昧な用語を明確化
3. 用語集を作成・維持
4. コードに用語を反映

**例**:
```
用語: 「注文」
定義: 顧客が商品を購入する意思表示。
      注文 = 顧客 + 商品リスト + 配送先 + 支払い方法
状態: 作成中 → 確定 → 発送済 → 完了/キャンセル
```

---

### 境界づけられたコンテキスト

**概要**: 同じ用語でも文脈によって意味が異なる

**例**:
```
販売コンテキスト:
  「顧客」= 購入履歴、与信、購買傾向

配送コンテキスト:
  「顧客」= 配送先住所、在宅時間

サポートコンテキスト:
  「顧客」= 問い合わせ履歴、契約内容
```

## 7. チェックリスト

### ヒアリング前

- [ ] ステークホルダーは特定できているか？
- [ ] ヒアリングの目的は明確か？
- [ ] 既存資料（業務フロー図等）を確認したか？
- [ ] 質問リストを準備したか？

### ヒアリング中

- [ ] 「なぜ」を十分に深掘りしたか？
- [ ] 具体例を引き出したか？
- [ ] 前提条件を確認したか？
- [ ] 例外ケースを確認したか？
- [ ] 理解を復唱して確認したか？

### ヒアリング後

- [ ] 議事録を作成したか？
- [ ] 不明点をリストアップしたか？
- [ ] 次のアクションは明確か？
- [ ] ステークホルダーにレビューを依頼したか？

### 要件定義書レビュー

- [ ] 全てのユーザーストーリーは INVEST を満たすか？
- [ ] 受入条件は具体的かつテスト可能か？
- [ ] 非機能要件は定量化されているか？
- [ ] 優先順位は明確か？
- [ ] スコープ外は明記されているか？
- [ ] 前提条件・制約は記載されているか？

## 8. テンプレート

### ユーザーストーリーテンプレート

```markdown
## US-{番号}: {タイトル}

**ストーリー**:
{役割}として、{機能}したい。なぜなら{価値}だから。

**受入条件**:
- [ ] {条件1}
- [ ] {条件2}
- [ ] {条件3}

**補足**:
- 前提条件: {前提}
- 除外事項: {スコープ外}
- 関連: US-{番号}

**非機能要件**:
- 性能: {要件}
- セキュリティ: {要件}
```

### 非機能要件テンプレート

```markdown
## NFR-{番号}: {カテゴリ} - {タイトル}

**要件**:
{定量的な要件記述}

**計測方法**:
{どのように検証するか}

**優先度**: Must / Should / Could

**関連 US**: US-{番号}
```
