# 設計原則・思想ガイド

## 概要

設計原則は「新しい・古い」ではなく「有用かどうか」で判断する。
このガイドは普遍的な設計原則をまとめ、実践的な適用指針を提供する。

---

## 1. SOLID 原則

### S - Single Responsibility Principle (単一責任の原則)

**定義**: クラスを変更する理由は一つであるべき

**実践的解釈**:
- 「変更理由」= ステークホルダー（アクター）
- 同じアクターに対する機能は同じモジュールに
- 異なるアクターの要求は分離

**適用例**:
```
悪い例: UserService が認証、プロフィール、通知を全て担当
良い例: AuthService, ProfileService, NotificationService に分離
```

**注意**: 過度な分離は複雑性を増す。変更頻度と変更理由を観察して判断。

---

### O - Open/Closed Principle (開放閉鎖の原則)

**定義**: 拡張に対して開き、修正に対して閉じる

**実践的解釈**:
- 新機能追加時に既存コードを変更しない
- 抽象化とポリモーフィズムで実現
- Strategy パターン、Plugin アーキテクチャ

**適用例**:
```
悪い例: if-else の連鎖で支払い方法を分岐
良い例: PaymentStrategy インターフェースで拡張可能に
```

**注意**: 全てを抽象化する必要はない。変更が予測される箇所のみ。

---

### L - Liskov Substitution Principle (リスコフの置換原則)

**定義**: 派生型は基底型と置換可能であるべき

**実践的解釈**:
- サブクラスは親の契約を守る
- 事前条件を強めない、事後条件を弱めない
- 「is-a」関係の厳密な適用

**典型的な違反**:
```
Square extends Rectangle  // 正方形は長方形ではない（振る舞いが異なる）
```

---

### I - Interface Segregation Principle (インターフェース分離の原則)

**定義**: クライアントに不要なメソッドへの依存を強制しない

**実践的解釈**:
- 「太った」インターフェースを避ける
- 役割ごとに小さなインターフェースに分割
- クライアントは必要なインターフェースのみ実装

**適用例**:
```
悪い例: IWorker { work(), eat(), sleep() }
良い例: IWorkable { work() }, IFeedable { eat() }
```

---

### D - Dependency Inversion Principle (依存性逆転の原則)

**定義**: 上位モジュールは下位モジュールに依存しない。両者は抽象に依存する。

**実践的解釈**:
- 具象ではなくインターフェースに依存
- 依存性注入 (DI) で実現
- テスト容易性の向上

**レイヤー構造**:
```
Domain (中心) ← Application ← Infrastructure (外側)
依存の向きは外から内へ
```

---

## 2. DRY / WET / AHA

### DRY (Don't Repeat Yourself)

**定義**: 全ての知識はシステム内で単一の明確な表現を持つべき

**誤解**: 「コードの重複を避ける」ではない
**正しい理解**: 「知識・ロジックの重複を避ける」

**適用すべき場面**:
- ビジネスロジックの重複
- 設定・定数の重複
- 検証ルールの重複

---

### WET (Write Everything Twice)

**定義**: 2回までは重複を許容する

**理由**:
- 早すぎる抽象化を防ぐ
- 本当のパターンが見えるまで待つ
- 間違った抽象化より重複の方がマシ

---

### AHA (Avoid Hasty Abstractions)

**定義**: 性急な抽象化を避ける

**Rule of Three**:
1. 1回目: そのまま書く
2. 2回目: 重複を認識するが、まだ抽象化しない
3. 3回目: パターンが明確になったら抽象化を検討

**判断基準**:
- 重複箇所は本当に同じ理由で変更されるか？
- 抽象化のコストは重複のコストより低いか？

---

## 3. シンプルさの原則

### KISS (Keep It Simple, Stupid)

**定義**: シンプルさを最優先する

**実践**:
- 最もシンプルな解決策から始める
- 複雑さは必要になってから追加
- 「賢い」コードより「明快な」コード

**指標**:
- 新人が理解できるか？
- 3ヶ月後の自分が理解できるか？

---

### YAGNI (You Aren't Gonna Need It)

**定義**: 必要になるまで作らない

**適用**:
- 「将来必要になるかも」で機能を追加しない
- 現在の要件を満たす最小限の実装
- 拡張性は必要になってから追加

**例外**:
- セキュリティ基盤
- ロギング・監視基盤
- 後から変更が困難な設計決定

---

### Worse is Better

**定義**: シンプルで不完全な解は、複雑で完全な解より優れることがある

**理由**:
- 早くリリースできる
- 理解しやすい
- 修正しやすい
- 実際のフィードバックを得られる

---

## 4. 関心の分離

### SoC (Separation of Concerns)

**定義**: 異なる関心事は異なるモジュールに分離する

**レイヤー例**:
```
Presentation Layer: UI/API
Application Layer: ユースケース
Domain Layer: ビジネスロジック
Infrastructure Layer: 外部システム連携
```

**横断的関心事の扱い**:
- ロギング、認証、トランザクション
- AOP (Aspect-Oriented Programming)
- ミドルウェア

---

### LoD (Law of Demeter / 最小知識の原則)

**定義**: 直接の友人とだけ会話する

**ルール**:
- 自分自身のメソッド
- 引数として渡されたオブジェクト
- 自分が生成したオブジェクト
- 自分のフィールド

**違反例**:
```
user.getAddress().getCity().getName()  // NG: チェーン呼び出し
user.getCityName()  // OK: 委譲メソッド
```

---

### POLA (Principle of Least Astonishment)

**定義**: 最も驚きが少ない動作をする

**実践**:
- 命名は動作を正確に表す
- 副作用を最小限に
- 慣例に従う
- 暗黙の動作を避ける

---

### CoC (Convention over Configuration)

**定義**: 設定より規約

**利点**:
- 設定ファイルの削減
- 学習コストの低減
- 一貫性の確保

**例**:
- Rails: テーブル名は複数形、モデル名は単数形
- Next.js: pages/ ディレクトリがルーティング

---

## 5. アンチパターン

### Over-Engineering (過剰設計)

**症状**:
- 必要以上の抽象化レイヤー
- 使われない拡張ポイント
- 過度なパターン適用

**対策**:
- YAGNI を意識
- 「今」必要な最小限の設計
- 定期的なリファクタリング

---

### Resume Driven Development (履歴書駆動開発)

**定義**: 技術的に面白い・履歴書に書けるという理由で技術選定

**対策**:
- 要件ベースの技術選定
- チームのスキルセット考慮
- 運用コストの評価

---

### Premature Optimization (早すぎる最適化)

**Donald Knuth**:
> "Premature optimization is the root of all evil"

**正しいアプローチ**:
1. まず動くコードを書く
2. 計測する
3. ボトルネックを特定
4. そこだけ最適化

---

### Premature Abstraction (早すぎる抽象化)

**症状**:
- 1つしか実装がないインターフェース
- 使われない汎用性
- 複雑な継承階層

**対策**:
- Rule of Three
- 具象から抽象へ（逆ではない）
- 実際の変更パターンを観察してから抽象化

---

## 6. 技術的負債

### 4象限モデル (Martin Fowler)

|          | 慎重 (Prudent) | 無謀 (Reckless) |
|----------|----------------|-----------------|
| **意図的 (Deliberate)** | 「今はリリース優先、後で返済」 | 「設計する時間がない」 |
| **不注意 (Inadvertent)** | 「今ならもっと良い方法がわかる」 | 「レイヤリングって何？」 |

### 負債の管理

**良い負債**:
- 意図的 × 慎重
- 返済計画がある
- ビジネス価値とのトレードオフが明確

**悪い負債**:
- 無謀または不注意
- 返済計画がない
- 雪だるま式に増加

**返済戦略**:
- Boy Scout Rule: 触ったコードは少し改善して去る
- 定期的なリファクタリング Sprint
- 負債の可視化（コメント、チケット）

---

## 7. 判断フレームワーク

### いつ原則を適用するか

| 状況 | 推奨アプローチ |
|------|----------------|
| プロトタイプ | KISS, YAGNI 優先。動くものを早く |
| MVP | シンプルさ優先。負債は意図的に |
| 成長期 | リファクタリング。SOLID 適用 |
| 成熟期 | 安定性重視。変更は慎重に |

### 原則の優先順位

1. **動くこと** - まず機能すること
2. **正しいこと** - バグがないこと
3. **読みやすいこと** - 理解できること
4. **シンプルなこと** - 複雑でないこと
5. **拡張しやすいこと** - 変更が容易なこと

## 8. チェックリスト

### コードレビュー時

- [ ] 単一責任: このクラス/関数の責務は一つか？
- [ ] 重複: 同じ知識が複数箇所にないか？
- [ ] シンプルさ: より簡単な方法はないか？
- [ ] 必要性: この機能/抽象化は今必要か？
- [ ] 驚き: 動作は名前から予測できるか？
- [ ] 依存: 依存関係は適切か？

### 設計時

- [ ] 要件を満たす最小限の設計か？
- [ ] 変更が予測される箇所は拡張可能か？
- [ ] テスト可能な設計か？
- [ ] チームが理解できる複雑さか？
- [ ] 技術的負債は意図的かつ管理可能か？
