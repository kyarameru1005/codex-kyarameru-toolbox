# アーキテクチャ意思決定チェックリスト

## 1. 要件確認チェックリスト

### 機能要件
- [ ] ユースケースの洗い出し
- [ ] データモデルの概要
- [ ] 外部システム連携
- [ ] ユーザー数・規模の見積もり

### 非機能要件
- [ ] **可用性**: SLA目標 (99.9%? 99.99%?)
- [ ] **性能**: レスポンスタイム目標、スループット
- [ ] **スケーラビリティ**: 成長予測、ピーク負荷
- [ ] **セキュリティ**: 認証要件、データ保護要件
- [ ] **コンプライアンス**: GDPR、PCI-DSS、等

### 制約条件
- [ ] 予算 (初期 / 月額運用)
- [ ] 期限 / マイルストーン
- [ ] チームスキル / 人数
- [ ] 既存システムとの統合要件

---

## 2. 技術選定チェックリスト

### 選定前の確認
- [ ] 複数の選択肢を検討したか？
- [ ] 最新情報を調査したか？ (WebSearch)
- [ ] チームに経験者はいるか？
- [ ] 学習コストを見積もったか？

### 比較観点
- [ ] **性能**: ベンチマーク、実績
- [ ] **エコシステム**: ライブラリ、ツール、コミュニティ
- [ ] **運用**: 監視、デバッグ、アップグレード
- [ ] **採用市場**: 人材の採用しやすさ
- [ ] **ライセンス**: OSS? 商用?
- [ ] **長期サポート**: LTS、EOL予定

### 決定後の記録
- [ ] ADR を作成したか？
- [ ] トレードオフを明示したか？
- [ ] ステークホルダーの合意を得たか？

---

## 3. アーキテクチャ設計チェックリスト

### 基本原則
- [ ] **単一責任**: 各コンポーネントの責務は明確か？
- [ ] **疎結合**: 依存関係は最小化されているか？
- [ ] **高凝集**: 関連する機能はまとまっているか？

### 可用性・信頼性
- [ ] 単一障害点 (SPOF) はないか？
- [ ] 障害時の復旧手順は明確か？
- [ ] バックアップ・リストア戦略はあるか？

### スケーラビリティ
- [ ] ボトルネックは特定されているか？
- [ ] 水平スケーリングは可能か？
- [ ] キャッシュ戦略は検討したか？

### セキュリティ
- [ ] 認証・認可は適切か？
- [ ] 機密データの保護は十分か？
- [ ] 入力検証は実装されるか？
- [ ] セキュリティ監査ログは取れるか？

### 運用性
- [ ] ログ・メトリクス・トレースは取得できるか？
- [ ] アラート設定は適切か？
- [ ] デプロイ・ロールバック手順は明確か？

## 4. ADR テンプレート

```markdown
# ADR-XXX: {タイトル}

## Status
Proposed | Accepted | Deprecated | Superseded by ADR-YYY

## Date
YYYY-MM-DD

## Context
- なぜこの決定が必要になったか
- 現状の課題
- 関連する制約条件

## Decision
- 何を決定したか
- 選択した選択肢

## Alternatives Considered
| 選択肢 | メリット | デメリット |
|--------|----------|------------|
| A | ... | ... |
| B | ... | ... |

## Consequences
### Positive
- 良い結果

### Negative
- 悪い結果、受け入れるトレードオフ

### Risks
- リスクと軽減策

## References
- 関連ドキュメント、調査結果
```

## 5. レビューチェックリスト

### 設計レビュー時
- [ ] 要件との整合性は取れているか？
- [ ] トレードオフは明示されているか？
- [ ] 代替案は検討されたか？
- [ ] セキュリティレビューは実施したか？
- [ ] コスト見積もりは妥当か？

### 実装後レビュー
- [ ] 設計通りに実装されているか？
- [ ] 性能要件を満たしているか？
- [ ] 運用手順書は整備されているか？
- [ ] 監視・アラートは設定されているか？
