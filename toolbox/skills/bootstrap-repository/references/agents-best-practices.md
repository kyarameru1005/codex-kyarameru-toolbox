# AGENTS.md Best Practices

## 目的

`AGENTS.md` は「AIエージェントが迷わず実行判断できるリポジトリ契約」を提供する。

## 設計原則

- 優先順位を固定する（例: 正確性 > 安全性 > 速度）。
- 抽象語を避け、条件と行動を明示する。
- 実行可能なコマンドを明記する（test/lint/typecheck/build）。
- 変更可能範囲・変更注意範囲・禁止事項を分けて記載する。
- Done条件と報告フォーマットを固定する。
- グローバルルールとリポジトリ固有ルールを分離する。
- 出力の型を固定する（例: 目的/変更点/検証結果）。
- 変化しやすい現状説明は `AGENTS.md` の正本にしない。
- `AGENTS.md` から分離できる詳細は `docs/` 配下へ正本を作る。

## 推奨セクション

1. 目的と優先順位
2. 応答ルール
3. 実行ルール（検証コマンド含む）
4. リポジトリ構造の扱い方針
5. 変更可能範囲 / 変更注意範囲 / 禁止事項
6. Done条件
7. Report format
8. Git / 安全性
9. ログ運用
10. 命名規則（必要なら構造ルール）

## リポジトリ構造の書き方

- `AGENTS.md` は原則として詳細なリポジトリ構造の正本にしない。
- 詳細構造の正本は `docs/repository-layout.md` に固定する。
- `AGENTS.md` にはディレクトリツリーを載せず、構造の参照先と変更判断に必要な高レベル方針だけを書く。
- 高レベル方針に留める場合は、`docs/` `scripts/` `toolbox/` などの責務を短く示す。
- `docs/repository-layout.md` は詳細構造の正本として、実ディレクトリに追従して更新する。
- `AGENTS.md` と `docs/repository-layout.md` の間で、参照先や責務の説明が矛盾してはならない。

高レベル参照の例:

- `リポジトリ構造の詳細は docs/repository-layout.md を参照する。本書では変更判断に必要な方針のみ扱う。`

## docs への分離方針

- `AGENTS.md` に置くべきでない詳細手順、詳細一覧、現状説明は `docs/` 配下へ分離する。
- 分離先は用途が分かる `kebab-case` の文書名を使う。
- 例:
  - コマンド一覧は `docs/command-reference.md`
  - 報告詳細テンプレートは `docs/reporting-guide.md`
  - 変更範囲の詳細運用は `docs/change-scope.md`
  - 構造の正本は `docs/repository-layout.md`
- `AGENTS.md` には分離先文書の要約を複製せず、参照先と判断方針だけを書く。

## repository-layout.md の推奨構成

- `docs/repository-layout.md` には `目的` `構成` `管理対象` `Git 管理しないもの` `変更時の確認` を含める。
- `構成` は `tree` 風のツリーを使い、各行の横に責務を短く書く。
- `管理対象` には正本ファイル、配備対象、運用スクリプト、テスト、主要ドキュメントを含める。
- `Git 管理しないもの` には生成物、キャッシュ、ローカル状態、追跡しない一時ファイルを列挙する。
- `変更時の確認` には、構造変更時に追随が必要なファイルや検証手順を書く。
- 生成後は `bash toolbox/skills/bootstrap-repository/scripts/check-repository-layout.sh docs/repository-layout.md AGENTS.md` で最低限の整合を確認する。

テンプレート例:

````md
# Repository Layout

この文書は、リポジトリ内の主要ディレクトリの責務と Git 管理方針を定義する。

## 構成

```text
.
├── docs/              # 仕様、運用メモ、レポート
├── scripts/           # 実行スクリプト
└── toolbox/           # 配備対象資材
```

## 管理対象

- `AGENTS.md`: リポジトリ運用ルールの正本。
- `docs/`: 構造と仕様の正本。

## Git 管理しないもの

- `toolbox/cache/`
- `*.egg-info/`

## 変更時の確認

- 配備対象を増やした場合は `scripts/install.py` と関連テストを更新する。
````

## 曖昧語の例（避ける）

- 適宜
- 必要に応じて
- 可能であれば
- 状況に応じて

## 良い記述例

- 悪い例: 「適宜テストする」
- 良い例: 「変更を加えたら `python3 -m pytest -q` を実行し、結果を報告する」

- 悪い例: 「必要に応じて確認する」
- 良い例: 「push / PR作成 / GitHub設定変更の前に実施内容をユーザーへ明示する」

## 変更可否範囲の書き分け例

- 変更可能: `docs/` は仕様更新のため編集可。`scripts/` は検証スクリプト追加可。
- 変更注意: `.github/workflows/` はCI影響があるため、理由と検証方法の記載を必須にする。
- 禁止: `.env` 編集、秘密情報の記録、`git reset --hard` の独断実行、`main` 直接push。

## Done条件テンプレート

```md
## Done条件
- 目的に対する変更が完了している
- 関連差分を確認している
- test/lint/typecheck のうち定義済みコマンドを実行している
- 実行できない検証は理由を記載している
- 変更内容と検証結果を報告している
```

## Report formatテンプレート

```md
## Report format
- 変更したファイル
- 変更内容
- 実行した検証コマンド
- 検証結果
- 残課題
- 人間が確認すべき点
```

## 最低完了条件

- 必須セクションがすべて存在する。
- 曖昧語が含まれない。
- Done条件が明文化されている。
- 命名規則とログ方針が明文化されている。
