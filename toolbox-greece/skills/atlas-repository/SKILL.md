# atlas-repository

リポジトリ全体の見通しを保ち、ファイルやディレクトリの置き場所を一定にするために使う。

## 重視すること

- 既存のディレクトリ責務を優先する
- 新しい置き場所を増やす前に既存構成で表現できるか確認する
- 移動、削除、リネームは影響範囲を確認してから行う
- ドキュメント、テスト、スクリプト、設定原本の境界を混ぜない
- 整理のための整理にならないよう、目的と効果を明確にする

## 基本方針

- `docs/` は運用、設計、調査、報告書、リポジトリ構造の正本文書を置く。
- `tests/` は自動テストを置く。
- `scripts/` は再利用する実行スクリプトを置く。
- プロジェクト固有ディレクトリの責務は `README.md` や `docs/distribution/repository-layout.md` で確認する。

## 手順

1. `README.md`、`docs/distribution/repository-layout.md`、対象ディレクトリの README を確認する。
2. 現在のファイル配置を `find` や `rg --files` で確認する。
3. 対象ファイルごとに「役割」「読者」「更新頻度」「実行されるか」を整理する。
4. 既存ディレクトリの責務に当てはまるか確認する。
5. 当てはまらない場合だけ、新しいディレクトリや分類を提案する。
6. 移動やリネームが必要な場合は、参照元、テスト、README、docs への影響を確認する。
7. 変更する場合は、最小差分で配置と関連文書を揃える。
8. 変更後に `python3 -m pytest -q` など関連する検証を実行する。

## 判断基準

- 置き場所に迷う場合は、まず「誰が読むか」で決める。
- 実行されるものは `scripts/` または実装側へ置く。
- 人が読む判断記録や報告書は `docs/` へ置く。
- テストからしか使わない補助データは `tests/` 配下へ置く。
- 複数箇所から参照される説明は README より `docs/` を優先する。
- 一時的なログ、キャッシュ、実行時データ、秘密情報は追加しない。

## 整理時の注意

- 既存変更がある場合は、ユーザーの変更を巻き戻さない。
- `cwd` 外の変更や破壊的操作は事前確認する。
- ファイル移動は、移動先だけでなく参照更新まで含めて判断する。
- 大量整形や無関係な命名変更は避ける。
- 構造変更後に README や `docs/` が古くなった場合は更新する。

## 出力

- 現在の構造の短い評価
- 整理対象と理由
- 推奨する置き場所や命名
- 実施する最小変更
- 影響範囲と注意点
- 実行した検証と結果
