# skills

このディレクトリは、`toolbox-greece` で使うスキルを置く場所です。
ここでいうスキルは、サブエージェントそのものではなく、「どう進めるか」の作業パターンです。

この toolbox では、次のように役割を分けます。

- エージェント名: 神名
- スキル名: 神話の道具・場所・概念

つまり、

- `Zeus` や `Hermes` は「誰が担当するか」
- `aegis-review` や `labyrinth-explore` は「どの型で進めるか」

を表します。

## スキル一覧

### `aegis-review`

レビュー用スキルです。
実装後の差分を見て、バグ、回帰、危険な仮定、保守性リスクを短く洗い出します。

名前の由来:
`aegis` はギリシャ神話の防具・盾で、防御的にリスクを見つけるレビューの役割に合わせています。

向いている場面:

- 実装後のセルフレビュー
- 他のエージェントが作った差分確認
- セキュリティ寄りの不安がある変更

組み合わせやすい相手:

- `Athena`
- `Ares`

### `labyrinth-explore`

事前調査用スキルです。
関連ファイル、処理フロー、依存関係、影響範囲を整理します。

名前の由来:
`labyrinth` は迷宮を意味し、複雑なコード構造の中で経路や影響範囲を探す調査の役割に合わせています。

向いている場面:

- どこを触るべきか分からない
- 影響範囲が広そう
- 実装前に構造理解が必要

組み合わせやすい相手:

- `Hermes`
- `Zeus`

### `forge-implement`

実装用スキルです。
変更対象がはっきりしているときに、最小差分で安全に進めるための型です。

名前の由来:
`forge` は鍛冶場を意味し、方針が固まったものを実際のコードや設定として形にする役割に合わせています。

向いている場面:

- 修正対象ファイルが分かっている
- 小さな機能追加や設定変更
- 既存方針に沿った実装

組み合わせやすい相手:

- `Hephaestus`

### `oracle-design`

設計判断用スキルです。
実装前に、構造、責務分離、選択肢比較、方針整理を行います。

名前の由来:
`oracle` は神託を意味し、実装前に進むべき方針や選択肢を見極める役割に合わせています。

向いている場面:

- どの構成にするべきか迷う
- 役割分担を見直したい
- 実装前に設計意図を固めたい

組み合わせやすい相手:

- `Daedalus`
- `Zeus`

### `gauntlet-verify`

検証用スキルです。
何を確認すべきかを整理し、テストやチェックの結果をまとめます。

名前の由来:
`gauntlet` は試練や篭手を意味し、変更をテストやチェックに通して品質を確認する役割に合わせています。

向いている場面:

- 実装後の確認
- どのテストを回すべきか決めたい
- 結果を PASS / FAIL / 未確認 で整理したい

組み合わせやすい相手:

- `Themis`
- `Athena`

### `chronicle-docs`

文書化用スキルです。
README、設計メモ、変更理由、記録整理を分かりやすくまとめます。

名前の由来:
`chronicle` は年代記や記録を意味し、作業内容や判断理由を後から読める文書として残す役割に合わせています。

向いている場面:

- README を書く
- 設計意図を残す
- 作業ログや振り返りを整理する

組み合わせやすい相手:

- `Apollo`
- `Chronos`

### `argo-git-flow`

Git 作業フロー用スキルです。
作業前のブランチ切り替え、ブランチ作成、ステージング、コミット、プッシュ、PR 作成を安全に進めます。

名前の由来:
`argo` はギリシャ神話の船で、変更をブランチから push、PR まで安全に運ぶ流れに合わせています。

向いている場面:

- 作業開始前に正しいブランチへ移動する
- 変更を確認して必要なファイルだけコミットする
- リモートへ push して PR を作る

組み合わせやすい相手:

- `Zeus`
- `Ares`

### `atlas-repository`

リポジトリ構造整理用スキルです。
ディレクトリ責務、ファイル配置、命名、docs/tests/scripts などの境界を確認し、構造を一定に保ちます。

名前の由来:
`atlas` は世界を支える巨神の名で、リポジトリ全体の構造を支え、見通しを保つ役割に合わせています。

向いている場面:

- ファイルの置き場所に迷う
- docs、tests、scripts などの境界を揃えたい
- リポジトリ整理前に影響範囲を確認したい

組み合わせやすい相手:

- `Hermes`
- `Daedalus`
- `Hephaestus`

### `zeus-orchestrate`

オーケストレーション発火用スキルです。
Zeus を入口にして、目的、前提、完了条件を確認し、必要な役割分担と進行計画を明示します。

名前の由来:
`Zeus` はこの toolbox の全体判断役で、作業開始時の指揮を明示的に発火する役割に合わせています。

向いている場面:

- 作業開始時に役割分担を明示したい
- Hermes、Daedalus、Hephaestus などを使うか判断したい
- 複雑な作業でチェックリストを作って進めたい

組み合わせやすい相手:

- `Zeus`
- `Hermes`
- `Daedalus`
- `Hephaestus`

### `muse-interface`

UI/ビジュアル設計の親スキルです。
画面レイアウト、情報設計、視覚階層、状態表現、一貫性、アクセシビリティといった共通の UI 構造を決めます。

名前の由来:
`muse` は芸術を司る女神たちで、見た目と体験の設計を担うスキル群の接頭辞に合わせています。

向いている場面:

- 実装前に UI/UX の方針を固めたい
- 画面構成や状態表現を整理したい
- 見た目の系統スキルと組み合わせて土台を決めたい

組み合わせやすい相手:

- `Aphrodite`
- 見た目の系統スキル（`muse-modern` / `muse-luxury` / `muse-minimal` / `muse-bold` / `muse-avantgarde`）

### `muse-modern` / `muse-luxury` / `muse-minimal` / `muse-bold` / `muse-avantgarde`

見た目のトーンを系統ごとに与えるデザイン系統スキルです。
`muse-interface` で決めた共通構造の上に、配色・タイポ・余白・装飾・モーションの方向を重ねます。

- `muse-modern`: クリーン・フラット・余白多めのモダンなトーン
- `muse-luxury`: 深色・金属アクセント・静けさの高級感トーン
- `muse-minimal`: 引き算・モノクローム寄りのミニマルなトーン
- `muse-bold`: 高コントラスト・太いタイポの大胆なトーン
- `muse-avantgarde`: 先鋭・実験的な最先端トーン

名前の由来:
`muse` 系統で世界観を揃え、神話語 + 見た目語で系統を区別しています。

向いている場面:

- UI の見た目を特定の方向に寄せたい
- 系統を1つ選んで一貫したトーンを与えたい

組み合わせやすい相手:

- `muse-interface`（親）
- `Aphrodite`

名前の由来:
`pandora` は災いが飛び出す箱の逸話に由来し、障害を制御下で扱い学びを得る役割に合わせています。

向いている場面:

- 耐障害性を検証する実験を設計したい
- 障害モードと安全策（中断・ロールバック）を整理したい

組み合わせやすい相手:

- `Khaos`
- `Themis`

組み合わせやすい相手:

## 基本の使い分け

迷ったときは、次の流れで考えると分かりやすいです。

1. まず調査が必要かを判断する
2. 設計判断が必要なら実装前に止まる
3. 実装後はレビューか検証を入れる
4. 説明が必要なら最後に文書化する

対応関係は次のとおりです。

- 調査: `labyrinth-explore`
- 設計: `oracle-design`
- UI/ビジュアル設計: `muse-interface`（親） + 見た目の系統スキル
- 実装: `forge-implement`
- レビュー: `aegis-review`
- 検証: `gauntlet-verify`
- 耐障害性検証: `pandora-chaos`
- 文書化: `chronicle-docs`
- Git 作業: `argo-git-flow`
- リポジトリ整理: `atlas-repository`
- 権限管理: `cerberus-permissions`
- 他 AI との会話: `agora-dialogue`
- オーケストレーション: `zeus-orchestrate`

## 典型的な組み合わせ

### 小さい修正

- `Zeus` -> `forge-implement` -> `aegis-review` または `gauntlet-verify`

### 影響範囲が不明な修正

- `Zeus` -> `labyrinth-explore` -> `forge-implement` -> `gauntlet-verify`

### 設計判断を含む修正

- `Zeus` -> `labyrinth-explore` -> `oracle-design` -> `forge-implement` -> `aegis-review` -> `gauntlet-verify`

### 文書中心の作業

- `Zeus` -> `chronicle-docs`

### UI/ビジュアル設計

- `muse-interface`（共通の UI 構造）で土台を決め、見た目の系統スキルを併用する。
- 例: `muse-interface` + `muse-luxury`（高級感に寄せる場合）
- 系統は基本どれか1つを選び、混在させないことで一貫性を保つ。

## デザイン系統スキル（`muse-*`）

UI/ビジュアル設計は「共通の構造」と「見た目のトーン」を分けて扱います。

- 親（共通の構造）: `muse-interface`
  - レイアウト・情報設計・状態表現・一貫性・アクセシビリティを決める。
- 系統（見た目のトーン）: `muse-modern` / `muse-luxury` / `muse-minimal` / `muse-bold` / `muse-avantgarde`
  - 配色・タイポ・余白・装飾・モーションの方向を系統ごとに与える。

使うときは「親で構造を決め、系統で見た目を寄せる」の順で組み合わせます。

## 命名ルール

この toolbox では、世界観を壊さないためにスキル名を神話の道具・場所・概念で統一します。
ただし、名前だけで意味が分からなくならないよう、神話語 + 実務語の形を保ちます。

例:

- `aegis-review`
- `labyrinth-explore`
- `forge-implement`
- `zeus-orchestrate`

この形なら雰囲気を保ちながら、初見でも役割を推測しやすくなります。

同じ系統のスキル群は共通の接頭辞でまとめます。
例: UI/ビジュアル設計の `muse-*`（`muse-interface`, `muse-modern`, `muse-luxury` …）。

## 追加するときの基準

新しいスキルを増やす前に、まず次を確認します。

- 既存スキルのどれでも代用できないか
- 新しいスキルに独立した作業パターンがあるか
- 名前だけで役割をある程度推測できるか

スキルは増やしすぎると使い分けが難しくなるので、最小セットを保つ方が運用しやすいです。
