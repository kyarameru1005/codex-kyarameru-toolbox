# toolbox-greece

`toolbox-greece` は、ギリシャ神話をモチーフにした Codex 用 toolbox です。
人間が 1 人で全部を抱え込まず、役割ごとにサブエージェントへ分担しやすいように構成しています。

このディレクトリは、Codex の設定原本そのものではなく、ギリシャ神話テーマで具体化した 1 つの toolbox です。
世界観のために神々の名前を使っていますが、目的は雰囲気作りではなく、責務分離を分かりやすくすることです。

## この toolbox の考え方

この toolbox では、1 つのエージェントに調査、設計、実装、レビュー、検証を全部やらせない方針を取ります。
代わりに、役割ごとにエージェントを分けて、メインの Codex が全体判断と分配を担当します。

狙いは次の 3 つです。

- 誰が何を担当したかを明確にする
- 調査不足のまま実装に進む事故を減らす
- 実装後に別視点でレビューや検証を入れやすくする

## ディレクトリ構成

- `AGENTS.md`
  この toolbox での共通ルールです。応答言語、変更範囲、破壊的操作の扱いを短く定義しています。

- `config.toml`
  Codex 全体の基本設定です。現在は最小構成で、モデルと推論強度だけを定義しています。

- `agents/`
  サブエージェント定義を置くディレクトリです。各 `*.toml` に役割、モデル、権限、行動指針を書きます。

- `skills/`
  再利用しやすい作業パターンを置くディレクトリです。この toolbox では神話の道具・場所・概念を名前に使います。

- `hooks/`, `prompts/`, `plugins/`, `mcp/`, `memories/`
  必要になったときに拡張するためのディレクトリです。現時点では空です。

## サブエージェント一覧

この toolbox には、次の役割が用意されています。

- `Zeus`
  全体判断を行うマネージャです。何を自分で処理し、何を別エージェントへ委譲するかを決めます。

- `Hermes`
  実装前の調査担当です。関連ファイル、処理フロー、依存関係、影響範囲を整理します。

- `Daedalus`
  設計担当です。責務分離、拡張性、アーキテクチャ判断を整理します。

- `Hephaestus`
  実装担当です。設計に沿ってコードや設定を変更します。

- `Athena`
  実装後レビュー担当です。正しさ、保守性、テスト不足、人間の理解リスクを確認します。

- `Themis`
  テスト・評価担当です。何を検証すべきかを整理し、結果を評価します。

- `Ares`
  セキュリティとリスク確認担当です。危険操作、権限過剰、秘密情報漏えいの可能性を見ます。

- `Apollo`
  ドキュメント担当です。README、仕様説明、設計メモ、研究資料を整えます。

- `Chronos`
  ログと振り返り担当です。実行記録、判断履歴、改善点を整理します。

詳しい考え方は [agents/README.md](/Users/ryukisato/deta_box/Project/kyarameru-tool-box/toolbox-greece/agents/README.md) を参照してください。

## スキル一覧

この toolbox では、エージェント名に神名を使い、スキル名には神話の道具・場所・概念を使います。
考え方としては、エージェントが「誰が担当するか」、スキルが「どの型で進めるか」を表します。

- `aegis-review`
  実装後レビュー用です。バグ、回帰、危険な仮定、保守性リスクを短く洗い出します。

- `labyrinth-explore`
  事前調査用です。関連ファイル、処理フロー、依存関係、影響範囲を整理します。

- `forge-implement`
  実装用です。対象が明確な変更を最小差分で進めます。

- `oracle-design`
  設計判断用です。構造、責務分離、選択肢比較、実装方針の整理に使います。

- `gauntlet-verify`
  検証用です。何を確認すべきかを整理し、テストやチェックの結果をまとめます。

- `chronicle-docs`
  文書化用です。README、設計メモ、変更理由、記録整理に使います。

この 6 個が、この toolbox の最小スキルセットです。

## オーケストレーションの基本

この toolbox では、メインの Codex がオーケストレーターとして振る舞います。
すべてを自分で処理するのではなく、必要な役割だけを呼び出して進めます。

基本の流れは次のとおりです。

1. まず `Zeus` がタスクを見て、軽い作業を自分で処理するか、別エージェントへ回すかを判断する。
2. 関連ファイルや影響範囲が不明なら `Hermes` を使って調査する。
3. 設計判断が必要なら `Daedalus` を使う。
4. 実装方針が固まったら `Hephaestus` で変更する。
5. 実装後は `Athena` や `Themis` で別視点の確認を入れる。
6. セキュリティや危険操作が気になる場合は `Ares` を挟む。
7. ドキュメント更新が必要なら `Apollo` と `chronicle-docs`、記録整理が必要なら `Chronos` を使う。

## 典型的な使い方

### 1. 小さい修正

単一ファイルで完結し、影響範囲が明確なら、`Zeus` が調査を省略してそのまま実装担当へ回す構成が向いています。

例:

- 軽微な設定修正
- 小さな文言変更
- 明確なバグ修正

流れ:

`Zeus` -> `Hephaestus` -> `Athena` または `Themis`

### 2. 影響範囲が読めない変更

どこを触るべきか曖昧な場合は、最初に `Hermes` を使って調査してから進めます。

例:

- リポジトリ構造がまだ分かっていない
- 依存関係が複雑
- 設定変更が複数箇所に波及しそう

流れ:

`Zeus` -> `Hermes` -> `Hephaestus` -> `Themis`

### 3. 設計判断を含む変更

責務分離や構成の見直しを伴う場合は、実装前に `Daedalus` を挟みます。

例:

- 新しい設定構造を追加する
- 既存の役割分担を見直す
- 拡張しやすい構成へ整理する

流れ:

`Zeus` -> `Hermes` -> `Daedalus` -> `Hephaestus` -> `Athena` -> `Themis`

### 4. ドキュメントや記録の整理

コードではなく説明が主目的なら、`Apollo` や `Chronos` を使います。

例:

- README の更新
- 設計意図の言語化
- 実験ログや振り返りの整理

流れ:

`Zeus` -> `Apollo` または `Chronos`

### 5. スキルとエージェントを組み合わせる

この toolbox では、役割だけでなく作業の型も分けます。
たとえば次のような対応が自然です。

- `Hermes` + `labyrinth-explore`
- `Daedalus` + `oracle-design`
- `Hephaestus` + `forge-implement`
- `Athena` または `Ares` + `aegis-review`
- `Themis` + `gauntlet-verify`
- `Apollo` または `Chronos` + `chronicle-docs`

この対応を固定ルールとして扱う必要はありませんが、迷ったときの基本線として使えます。

## 運用ルール

この toolbox では、次のルールを重視します。

- 推測より確認を優先する
- 変更は Codex を起動した `cwd` 配下に限定する
- 変更は必要最小限にとどめる
- `cwd` 外の変更や破壊的操作は事前確認する
- 秘密情報や実行時データは追加しない

短い版は [AGENTS.md](/Users/ryukisato/deta_box/Project/kyarameru-tool-box/toolbox-greece/AGENTS.md) にあります。

## 並列化するときの注意

サブエージェントを増やせば常に速くなるわけではありません。
並列化は、完全に独立した作業だけに限定するのが安全です。

特に次は避けた方がよいです。

- 同じファイルを複数エージェントで同時に編集する
- 調査前に複数の実装エージェントを先に走らせる
- 設計が固まる前に実装とレビューを並列で進める

逆に、次のような分け方は相性がよいです。

- 調査とドキュメント整理を分ける
- 実装後レビューとテスト整理を分ける
- セキュリティ確認を別役割に分ける

## 現在の状態

現時点では、`config.toml` は最小構成です。
そのため、この toolbox は「役割分担の考え方」と「エージェント定義」が中心で、プロファイル切替や詳細な runtime 設定はまだ絞り込んでいません。

- hooks や prompts の追加

## 使い始める人への目安

最初は全部を使おうとしない方が安全です。
まずは次の最小パターンから始めると扱いやすいです。

1. 判断は `Zeus`
2. 調査が必要なら `Hermes`
3. 実装は `Hephaestus`
4. 確認は `Athena` または `Themis`

この流れに慣れてから、設計担当の `Daedalus`、文書担当の `Apollo`、記録担当の `Chronos`、リスク担当の `Ares` を足していくと、運用を崩しにくくなります。
