# Benchmark Cases

toolbox の品質を確認するための共通ベンチマーク項目です。
各ケースは、入力、期待する動き、見る観点を固定して実行します。

## Case 1: 単純な文書修正

入力:

```text
README の説明文を1文だけ分かりやすくしてください。
```

期待:

- 変更範囲を小さく保つ
- 不要な調査や設計担当を使わない
- 検証結果を報告する

見る観点:

- Zeus の判断が過剰でないか
- 実装担当または文書化担当の出力が簡潔か
- 差分が依頼範囲に収まっているか

## Case 2: 影響範囲が不明な修正

入力:

```text
toolbox 適用時の dry-run 表示が分かりにくいので改善してください。
```

期待:

- 実装前に関連ファイルと影響範囲を調べる
- 実装対象を絞ってから変更する
- 検証コマンドを提案または実行する

見る観点:

- Hermes または調査スキルを使う判断があるか
- 調査結果が実装に接続しているか
- Themis が不足検証を拾えるか

## Case 3: 設計判断を含む設定追加

入力:

```text
新しい toolbox の種類を増やせるように、設定の置き場所を整理してください。
```

期待:

- すぐ実装せず、設計判断を整理する
- 既存構造との責務境界を確認する
- 採用しない案も短く説明する

見る観点:

- Daedalus または設計スキルを使う判断があるか
- 過剰設計になっていないか
- 実装担当へ渡せる粒度になっているか

## Case 4: 実装後レビュー

入力:

```text
この差分をレビューして、バグや不足している検証を指摘してください。
```

期待:

- 重要度順に問題を出す
- スタイルだけの指摘に偏らない
- 修正案または次の確認手順を示す

見る観点:

- Athena の指摘が具体的か
- Themis の検証観点が実行可能か
- 指摘数が多すぎないか

## Case 5: セキュリティリスク確認

入力:

```text
この変更に秘密情報や危険操作のリスクがないか確認してください。
```

期待:

- 秘密情報、cwd 外変更、破壊的操作を確認する
- リスクを重大度で分類する
- 危険な手順を提案しない

見る観点:

- Ares の判定が保守的すぎないか
- 対象ファイルと理由が明確か
- 安全な代替案があるか

## Case 6: 長い作業の再開メモ

入力:

```text
ここまでの作業を、途中再開できるように短くまとめてください。
```

期待:

- 目的、現在地、完了済み、未完了、次の一手を分ける
- 触ったファイルと検証結果を残す
- 推測と事実を混ぜない

見る観点:

- Chronos の記録が再開に使えるか
- 長すぎないか
- 次にやることが明確か
